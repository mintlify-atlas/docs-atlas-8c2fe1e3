El backend soporta dos modos de autenticación controlados por la variable `AUTH_MODE`. El modo se selecciona una vez y aplica a todos los inicios de sesión del sistema.

## Modos de autenticación

<CardGroup cols={2}>
  <Card title="AUTH_MODE=local" icon="flask">
    **Para pruebas y desarrollo**

    El backend valida las credenciales contra la tabla `usuarios` en la base de datos. Las contraseñas se almacenan con hash bcrypt.

    - Los usuarios legacy con contraseñas en texto plano son actualizados automáticamente a bcrypt en el primer inicio de sesión exitoso.
    - No requiere conexión a Active Directory.
    - El departamento del usuario es el registrado en la base de datos; no se consulta LDAP.
  </Card>

  <Card title="AUTH_MODE=ldap" icon="shield">
    **Para producción**

    El backend autentica las credenciales directamente contra Active Directory usando `ldapjs`. La contraseña nunca se almacena en la base de datos.

    - Requiere las variables `LDAP_*` correctamente configuradas.
    - Los usuarios se crean o actualizan en la BD automáticamente al iniciar sesión (función `syncUser`).
    - El departamento se resuelve desde los atributos LDAP del usuario.
  </Card>
</CardGroup>

## Flujo de autenticación LDAP

Cuando `AUTH_MODE=ldap`, el proceso de login sigue estos pasos:

1. El backend realiza un bind con la cuenta de servicio (`LDAP_SERVICE_DN` / `LDAP_SERVICE_PASS`).
2. Busca al usuario por `sAMAccountName` dentro de `LDAP_BASE_OU`.
3. Obtiene el DN completo del usuario desde el resultado de búsqueda.
4. Realiza un bind con el DN del usuario y la contraseña ingresada para verificar las credenciales.
5. Si el bind es exitoso, el usuario está autenticado.
6. Se ejecuta `syncUser`: si el usuario no existe en la BD, se crea; si ya existe, se actualiza su nombre completo (el departamento no se modifica).

## Sincronización de usuarios (`syncUser`)

En cada inicio de sesión exitoso (en ambos modos), el sistema ejecuta un upsert en la tabla `usuarios`:

- **Usuario nuevo:** se inserta con el login, nombre completo, email (`login@farmcorp.com.ec`), el departamento resuelto y rol 1 (rol base).
- **Usuario existente:** solo se actualiza el campo `nombres`. El departamento asignado es **inmutable** una vez establecido.

<Note>
  El departamento no se modifica en actualizaciones posteriores aunque el usuario cambie de OU en Active Directory. Para reasignar el departamento de un usuario es necesario modificarlo directamente en la tabla `usuarios` de la base de datos.
</Note>

## Resolución de departamentos

Cuando un usuario se autentica por primera vez en modo LDAP, el sistema debe determinar a qué departamento pertenece. La lógica está implementada en la función `buildDepartmentCandidates` de `auth.js`:

### Regla especial: Comercial

Si se cumple alguna de las siguientes condiciones, el usuario se asigna directamente al departamento **Comercial**:

- El DN del usuario contiene `ou=pdv` (pertenece a la OU de Puntos de Venta).
- El atributo `department` del usuario en LDAP es exactamente `servidor`.

### Resolución general

Para el resto de los usuarios, se construye una lista ordenada de candidatos:

1. Se normaliza el valor del campo `department` del atributo LDAP (minúsculas, sin tildes).
2. Se extraen todas las OUs del DN del usuario, en orden desde la más cercana al CN.
3. Se filtran los candidatos que coincidan con OUs genéricas no informativas: `oficina`, `usuario`, `usuarios`, `farmcorp`, `asistentes`.

El sistema busca en la tabla `departamentos` de la BD el primer candidato que coincida (por nombre normalizado). El primer match se usa como departamento del usuario.

### Ejemplo de resolución

```
DN: CN=Juan Perez,OU=Recursos Humanos,OU=usuarios,OU=farmcorp,DC=farmcorp,DC=com,DC=ec
department (LDAP): Recursos Humanos

Candidatos generados:
  1. "recursos humanos"  ← del campo department
  2. "recursos humanos"  ← de OU=Recursos Humanos (duplicado, ignorado)
  ← "usuarios" filtrado por ser OU genérica
  ← "farmcorp" filtrado por ser OU genérica

Resultado: se busca departamento con descripcion = 'Recursos Humanos' en la BD.
```

## Endpoint de consulta de departamento

```
GET /api/auth/departamento/:username
```

Este endpoint permite al frontend pre-llenar el campo de departamento en el formulario de login antes de que el usuario ingrese su contraseña.

- Primero busca al usuario en la base de datos (funciona en ambos modos y para usuarios ya registrados).
- Si no existe en la BD y `AUTH_MODE=ldap`, consulta Active Directory y aplica la misma lógica de resolución de departamento.
- Devuelve `id_departamento`, `departmentName` y `nombres` del usuario.

## Configuración de variables LDAP

<Note>
  El código de `ldap.js` resuelve la URL del servidor como `LDAP_URL || LDAP_HOST`. Puedes usar cualquiera de las dos variables; si defines ambas, `LDAP_URL` tiene prioridad.
</Note>

```env
AUTH_MODE=ldap
LDAP_HOST=ldap://SRVDCR000.farmcorp.com.ec
LDAP_SERVICE_DN=src_TI_ldap@farmcorp.com.ec
LDAP_SERVICE_PASS=TU_CONTRASEÑA_CUENTA_SERVICIO
LDAP_BASE_OU=ou=usuarios,ou=farmcorp,dc=farmcorp,dc=com,dc=ec
```

<Warning>
  La cuenta de servicio definida en `LDAP_SERVICE_DN` necesita permisos de **lectura** sobre los objetos de usuario en Active Directory. Específicamente, debe poder realizar búsquedas por `sAMAccountName` y leer los atributos `distinguishedName`, `displayName`, `cn`, `mail` y `memberOf`.

  Si la cuenta de servicio no tiene los permisos adecuados, todos los inicios de sesión fallarán con un error de bind administrativo.
</Warning>
